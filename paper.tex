\begin{abstract}
We present an agent-assisted, workload-adaptive implementation of the DeepSeek-V3 MoE kernel targeting the NVIDIA B200 (Blackwell, sm\_100a) GPU.
Our fused Triton pipeline exploits FP8 native tensor core dot products for 2$\times$ throughput in GEMM1, a non-atomic two-pass GEMM2-then-reduce architecture, FP16 intermediate buffers with scale-and-cast compensation, and multi-level bucket specialization across BLOCK\_M tiers.
For the largest traces, a hybrid Triton/CuTe path uses Triton for sparse MoE metadata and epilogues while delegating the grouped GEMM core to CuTe.
On 19 real-trace workloads ($T{=}1$ to $T{=}14{,}107$), the system achieves 19/19 correctness with a peak speedup of ${\sim}56\times$, large-$T$ speedups of ${\sim}13$--$15\times$, and a mean of ${\sim}54\times$ on reference Modal B200 sessions, as measured by CUPTI GPU-only kernel timing.
\end{abstract}

% ============================================================
\section{Introduction}

Mixture-of-Experts (MoE) architectures enable massive parameter counts with sub-linear compute growth.
DeepSeek-V3~\cite{deepseekv3} uses 256 routed experts (32 local per device), top-8 routing, hidden dimension 7168, and intermediate dimension 2048---yielding two FP8 block-scaled GEMMs per forward pass.

On NVIDIA B200, na\"ive per-expert execution wastes hardware: launch overhead dominates small traces, while large traces suffer from atomic scatter in reduction.

This work was developed for Track~A of the MLSys 2026 FlashInfer-Bench Challenge~\cite{flashinfer}, where scoring is the sum of CUPTI GPU kernel times across 19 workloads (tolerance $\text{atol}{=}1.0$, $\text{rtol}{=}0.3$, $\text{ratio}{=}0.9$).
CUPTI excludes CPU overhead, so optimization is driven by GPU kernel efficiency.
Our contributions:

\begin{enumerate}
\item A fused Triton~\cite{triton} pipeline eliminating per-expert launches;
\item Native FP8 tensor core dot in GEMM1, delivering 2$\times$ throughput over TF32;
\item Non-atomic GEMM2 with token-centric reduce, yielding +46\% on medium-$T$;
\item FP16 intermediate buffers with $\times$0.125/$\times$8.0 compensation ($-$50\% bandwidth);
\item Workload-aware dispatch, including singleton, tight-padding, isolated transition, and CuTe-assisted large-$T$ buckets.
\end{enumerate}

\section{System Architecture}

\subsection{Pipeline Overview}

Figure~\ref{fig:pipeline} shows the generic five-stage pipeline:
\textbf{(1)~Routing:} Triton DeepSeek-V3 no-aux routing (sigmoid, group top-4, global top-8, normalization) with one program per token.
\textbf{(2)~Sorting:} Tile-level parallel histogram with atomic offsets and padded row construction.
\textbf{(3)~GEMM1+SwiGLU:} Fused $W_1$/$W_3$ projections in a shared K-loop with \texttt{tl.dot(fp8, fp8)}.
\textbf{(4)~GEMM2:} Non-atomic down-projection writing to the \texttt{expert\_out} buffer.
\textbf{(5)~Token Reduce:} Each output token reads $K{=}8$ expert contributions, accumulates in FP32, writes bf16.

\end{figure}

\subsection{Dispatch and Specialization}

Workloads span four orders of magnitude, so a single tile shape is suboptimal.
We therefore organize execution around a default BLOCK\_M tiering rule plus a small number of explicit trace-specific overrides (Table~\ref{tab:dispatch}).
The main overrides are: singleton $T{=}1$, tight BLOCK\_M=8 for fragmented $T{=}53$--59, isolated $T{=}901$, and a hybrid Triton/CuTe path for the largest traces.
For $T{\ge}4096$, the dispatcher switches from static overlaunch to exact block-count launch using \texttt{total\_blocks} produced by sorting, avoiding ${\sim}30\%$ idle blocks.

\section{Key Optimizations}

\subsection{FP8 Native Tensor Core Dot}

Blackwell's \texttt{tcgen05.mma} supports native FP8$\times$FP8 dot with FP32 accumulation. We apply block-scale dequantization \emph{after} the dot:

\vspace{-2mm}
\begin{verbatim}
  partial = tl.dot(a, b)  # fp8 x fp8
  acc += partial * a_scale * b_scale
\end{verbatim}
\vspace{-2mm}

This yields ${\sim}2\times$ throughput vs.\ TF32, which requires FP8$\to$FP32 dequantization before the dot. GEMM1 stores activations and weights in \texttt{float8\_e4m3fn} with per-128 scaling and fuses $W_1$/$W_3$ with inline SwiGLU.

\subsection{Non-Atomic GEMM2 + Token-Centric Reduce}

Na\"ive \texttt{tl.atomic\_add} causes $6\times$ bandwidth amplification (12~B/elem FP32 RMW vs.\ 2~B bf16 store).
We split GEMM2 and reduction: GEMM2 writes non-atomic \texttt{expert\_out} (bf16 where enabled), then a token-centric reduce loads 8 contributions per token. For large $T$, 2D tiling cuts the reduce grid $>$10$\times$.
A/B testing confirmed this as one of the largest single gains, especially on medium-$T$ workloads where atomic contention previously dominated.

\subsection{FP16 Intermediate with Scale Compensation}

The Intermediate buffer $[\text{num\_padded}, 2048]$ dominates HBM traffic.
We scale by $\times$0.125 in GEMM1, store FP16, then compensate by $\times$8.0 in GEMM2.
For $T{<}32$, a \texttt{constexpr} branch falls back to FP32 to avoid SwiGLU outlier overflow.
A/B: mean +4.5\% (13/19 improved).

\subsection{Workload-Adaptive Bucket Specialization}

Column-major dispatch (\texttt{GROUP\_M=32/64}) improves L2 reuse by mapping consecutive blocks to different tokens of the same expert.
The $T{=}1$ path is a dedicated decode implementation: it bypasses generic histogram/offset metadata and uses two fused kernels (GEMM1+SwiGLU and GEMM2+reduce) around a fixed small workspace.
For fragmented $T{=}53$--59, BLOCK\_M=8 reduces padding and invalid-row compute. For $T{=}901$, an isolated path targets the transition regime.

\subsection{Routing, Sorting, and Autotuning Support}

Several supporting optimizations are smaller individually but important to the final system.
Routing uses a Triton kernel implementing group-filter then global top-K selection with pre-allocated workspaces; sorting uses tile-parallel histogram/offset construction and, for large traces, reuses the histogram structure in dispatch.
Autotuning remains workload-specific: GEMM1/GEMM2 tile shapes, \texttt{BLOCK\_K}, and stage depth are selected per regime rather than by a single global configuration.

\subsection{CuTe-Assisted Large-T GEMM}

For the largest traces, bottlenecks shift from launch/padding overhead to sustained grouped GEMM throughput.
We therefore use a hybrid path for $T{=}11{,}948$ and $T{=}14{,}107$: Triton handles sparse metadata and epilogues, while CuTe executes the grouped GEMM core.
We do not use CuTe for small/mid-$T$, where metadata and setup dominate.

\subsection{bf16 Expert Output Buffer}

The \texttt{expert\_out} buffer $[\text{MAX\_PADDED}, 7168]$ is stored in bf16 ($T{\ge}32$), saving 50\% write bandwidth.
bf16's $3.4{\times}10^{38}$ range avoids the fp16 overflow that caused \texttt{inf} on 3/19 workloads.
A/B: +0.4\% mean, +2.5\% at $T{=}14{,}107$.

\subsection{Compiler Hints and Cache Policy}

Two compiler hints yielded a combined +4.9\% mean speedup:
\textbf{(1)~Disable LLVM LSR:}
Setting \texttt{DISABLE\_LLVM\_OPT="disable-lsr"} avoids extra register pressure and spills in SwiGLU addressing.
\textbf{(2)~Asymmetric L2 eviction:}
Expert weights use \texttt{evict\_last}; token activations use \texttt{evict\_first}.

\subsection{2D Tiled Token Reduce}

The original 1D reduce launched one program per (token, N-block) pair, producing a large grid at $T{=}14{,}107$.
Switching to 2D $[\text{BLOCK\_T}, \text{BLOCK\_N}]$ tiling amortizes the per-token loop overhead across BLOCK\_T rows, cutting the grid $>$10$\times$ and improving occupancy.
A/B: +2.5--2.9\% on the largest traces.

\subsection{Deep Software Pipelining for GEMM2}

GEMM2's FP16 A-side incurs high HBM latency.
Extending autotuning to \texttt{num\_stages=6/7/8} overlaps compute with reads and yields +2.1\% mean (+10.1\% peak) on medium-$T$ workloads.

\section{Evaluation}

\textbf{Setup.}
We evaluate with the official FlashInfer-Bench CLI on NVIDIA B200 (sm\_100a).
Scoring uses CUPTI GPU kernel time over 19 real-trace workloads; CPU overhead is excluded.
All claims use same-session A/B testing with a ${\pm}2\%$ mean noise floor.

\subsection{Results}

Figure~\ref{fig:speedup} and Table~\ref{tab:results} summarize representative performance by workload regime.
\textbf{Correctness:} 19/19. \textbf{Best observed speedup:} ${\sim}56\times$. \textbf{Mean speedup:} ${\sim}54\times$.
Small and fragmented traces benefit most because the baseline pays repeated per-expert launch overhead while our fused path amortizes routing, sorting, and GEMM dispatch.
$T{=}1$ is best treated as a singleton specialization rather than part of the grouped-kernel trendline.
The two largest traces sustain ${\sim}13$--$15\times$ speedup in reference Modal sessions.

\subsection{Profiling Analysis}

NCU profiling reveals three bottleneck regimes (Figures~\ref{fig:breakdown}--\ref{fig:tflops}):
\textbf{Small/medium $T$ (7--80):} GEMM1 dominates because expert weights are repeatedly loaded for very few routed rows, with substantial padding waste.
\textbf{$T{=}901$:} the workload sits in a GEMM1/GEMM2 transition regime, motivating an isolated path.
\textbf{Large hybrid $T$ (11{,}948--14{,}107):} GEMM1 still takes the larger time share (2$\times$ the FLOPs from fused $W_1$/$W_3$), but GEMM2 is the efficiency bottleneck---its FP16 A-side prevents FP8 tensor core use, capping throughput well below peak. CuTe improves the grouped-GEMM core.
Token reduce remains lightweight across regimes.

\subsection{Measurement Noise}

Modal B200 shared instances exhibit ${\pm}2\%$ mean noise and ${\pm}15\%$ per-workload noise for small $T$, but $<$1\% for large $T$.
Cross-session drift reaches ${\pm}20{-}30\%$; all claims use same-session A/B testing with a 2\% mean threshold.
The contest tolerance ($\text{atol}{=}1.0$, $\text{rtol}{=}0.3$, 90\%) is ${\sim}100\times$ wider than our internal checks, enabling bf16 expert output.

\section{Tools \& Languages}

We implement in \textbf{Python}, using \textbf{Triton}~\cite{triton} for routing/sorting/GEMM/reduce kernels and \textbf{CuTe DSL} for the two largest grouped-GEMM workloads.
We evaluated raw CUDA, Tilelang, and CuTile, but retained Triton/CuTe because Triton's FP8 intrinsics and autotuner covered 17/19 workloads efficiently and CuTe cleanly handled the remaining two.

% ============================================================
\section{Human vs.\ Agent Contributions}

Human contributors led the profiling-driven strategy: they identified workload-adaptive dispatch as the organizing principle, decided when pure Triton had reached diminishing returns, proposed introducing CuTe DSL for the largest grouped-GEMM traces, and selected which experiments to keep or revert.
They also interpreted benchmark noise, made final correctness/risk decisions, and maintained the contest-facing solution.
The agent assisted by inspecting profiler traces, expanding human hypotheses into bounded Triton/CuTe experiments, comparing A/B results, and summarizing failed directions.
All final optimizations were human-reviewed against correctness, benchmark stability, and simplicity before inclusion.

% ============================================================
\section{Conclusion}

We presented a workload-adaptive MoE kernel system achieving ${\sim}56\times$ peak and ${\sim}54\times$ mean speedup on B200 through 23 rounds of profiling-driven optimization, combining FP8 dot, non-atomic GEMM2, FP16 intermediate, workload-aware dispatch, and CuTe-assisted large-$T$ GEMM.
Eight rejected approaches fall into precision failures and amortization failures where $K{=}2048$ does not justify setup cost.
